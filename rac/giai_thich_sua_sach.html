Cơ Chế Sửa Sự Sách Trong Hệ Thống

Project không có file SachController riêng biệt để sửa sách, vậy chức năng Sửa Sách hoạt động như thế nào?

📄 File: src/.../dao/SachRepository.java
@RepositoryRestResource(path="sach")
public interface SachRepository extends JpaRepository<Sach, Integer> {
    // ... các hàm find ...
}

Cơ chế Spring Data REST: Thay vì phải viết các hàm Controller phức tạp để Sửa dữ liệu (như truyền thống), bạn đã sử dụng Annotation tĩnh @RepositoryRestResource. Nó tự động phơi bày (expose) toàn bộ các Endpoints thao tác CRUD cơ bản ra ngoài trực tiếp từ Interface Repository.

Nghĩa là nếu Frontend gửi Yêu cầu HTTP kiểu PUT /sach/{id} hoặc PATCH /sach/{id} với cục dữ liệu JSON mới, Spring Boot sẽ tự động tóm lấy ID đó, tìm cuốn sách, đắp dữ liệu JSON mới đè lên, và lưu xuống DB mà không cần bạn phải rờ tay viết 1 dòng code logic nào cả.

📄 File: src/.../security/Endpoints.java
public static final String[] ADMIN_PUT_ENDPOINTS = { //123SuaSach
    "/sach/**",    //123SuaSach - Admin được phép PUT (sửa) sách
};

Bảo vệ đầu mút (Endpoints): Bởi vì Endpoints sửa sách là tự động lộ ra do Spring Data Rest, nên bạn định nghĩa một danh sách các đường dẫn cần được bảo mật trong class Endpoints. Mọi yêu cầu đi vào `/sach/...` qua phương thức PUT và PATCH đều được gom vào một hạm đội bảo vệ chống lại User bình thường.

📄 File: src/.../security/SecurityConfiguration.java
http.authorizeHttpRequests(config -> config
     // ...
     .requestMatchers(HttpMethod.PUT, Endpoints.ADMIN_PUT_ENDPOINTS).hasAnyAuthority("ROLE_ADMIN")
     .requestMatchers(HttpMethod.PATCH, Endpoints.ADMIN_PUT_ENDPOINTS).hasAnyAuthority("ROLE_ADMIN")
);

Xác Thực Quyền Hạn: Dòng code này khóa cứng 2 phương pháp là PUT và PATCH (2 phương thức đại diện cho việc Sửa/Cập nhật dữ liệu), và chỉ cho phép những người nào mang thẻ ROLE_ADMIN trong Token của họ đi qua cổng này.
